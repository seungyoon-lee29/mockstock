# 모의 주식게임 — 아키텍처 재설계 브레인스토밍 (풀스택 아키텍트 에이전트)

현재 코드 기준 요약: `web/`만 존재. mock 시세는 `globalThis` 싱글톤 + `setInterval`(`web/src/lib/market/priceSource.ts`), SSE는 Next route가 1초 폴링으로 흘려보내는 스텁(`web/src/app/api/stream/route.ts`), 클라이언트는 `usePrices.ts`가 EventSource 구독. DB는 Neon+drizzle 의존성만 있고 스키마 없음. `prevClose`는 `seedPrice`로 가짜 처리 중.

---

## 1. 아키텍처 옵션 3개

### 옵션 A — Vercel 단일 앱 + 서버 캐시 폴링

```
Finnhub REST ─┐
KIS REST ─────┤→ Next API /api/quotes (서버 캐시 2~3s TTL)
              │
브라우저 ──(2~3초 폴링, react-query refetchInterval)──→ Vercel Functions ──→ Neon
```

- **장점**: 배포 1개, 인프라 0개, 지금 스택 그대로. 무료.
- **단점**: "실시간"이 2~3초 지연 폴링. 틱 단위 깜빡임(토스 감성)이 안 나옴. Vercel 함수는 인스턴스별 메모리 분리로 캐시 일관성 문제(Upstash KV 붙이면 인프라 +1). "실시간 파이프라인 만들어봤다"는 어필 포인트가 죽음.
- **월 비용**: $0 / **운영 부담**: 최소

### 옵션 B — web(Vercel) + 실시간 워커(Railway/Fly) ← 추천, README 계획과 동일

```
Finnhub WS ──┐                        ┌─(SSE, CORS 직결)──→ 브라우저
KIS WS ──────┤→ worker (Node 상주)    │
             │   · 인메모리 last-price book
             │   · 틱 배치(500ms~1s) 팬아웃
             │   └─ GET /snapshot ←──(주문 체결가 조회)── web /api/orders
             │
브라우저 ──(주문/포트폴리오 REST)──→ web (Vercel) ──→ Neon (drizzle)
```

핵심: 브라우저의 SSE는 **워커에 직결**(Vercel 경유 안 함). Vercel은 인증·주문·포트폴리오 CRUD만.

- **장점**: 진짜 틱 스트림. 업스트림 WS 연결이 사용자 수와 무관하게 딱 1개씩(레이트리밋 안전). 체결가 조회(`/snapshot`)가 단일 소스라 서버 체결이 깔끔. 현재 mock 코드를 워커로 옮기면 키없음 모드도 그대로. 포트폴리오 어필 최대.
- **단점**: 배포 2개, WS 재연결·KIS 토큰 수명 관리 직접 구현. Render 무료 티어는 15분 슬립(첫 방문 50초 대기 — 데모에 치명적) → 유료 최소 플랜 권장.
- **월 비용**: Vercel $0 + Railway Hobby ~$5(또는 Fly ~$3) + Neon $0 → **~$5/월**
- **운영 부담**: 중간. 워커 상태는 인메모리뿐이라 죽으면 재시작하면 그만.

### 옵션 C — Supabase 올인 (Realtime Broadcast + DB + Auth)

```
Finnhub/KIS WS → 초소형 ingest 프로세스(어딘가 상주) → Supabase Realtime Broadcast
브라우저 ← supabase-js 채널 구독
web (Vercel) → Supabase Postgres + Supabase Auth
```

- **장점**: 팬아웃·인증·DB가 한 플랫폼. 재연결·멀티플렉싱을 supabase-js가 처리.
- **단점**: **ingest 프로세스는 여전히 어딘가 상주 필요** → 워커가 사라지지 않음. 무료 티어 Realtime 메시지 2M/월인데 1초 브로드캐스트만으로 월 ~2.6M → Pro $25/월 절벽. 벤더 락인.
- **월 비용**: $0 시작 → 쿼터 초과 시 $25/월 + ingest 호스팅 $3~5

**판정**: B. A는 데모 품질이 아깝고, C는 워커를 없애주지도 못하면서 쿼터 리스크만 추가. `shared/`는 별도 패키지 말고 타입 파일 공유 수준으로 시작.

## 2. 핵심 기술 결정 포인트

### 시세 파이프라인
- 워커가 업스트림 WS 2개(Finnhub, KIS)를 각 1커넥션 유지, 정규화해서 인메모리 `Map<key, Tick>` 갱신.
- 클라이언트 전달은 **SSE**(현 `usePrices.ts` 그대로 호환). 하향 단방향 + EventSource 자동 재연결 공짜 → WS로 갈 이유 없음.
- 틱은 **500ms~1s 배치**로 flush. 지금 mock이 이미 1초 배치라 클라이언트 코드 변경 없음.
- Finnhub 무료 심볼 ~50개 제한 → 유니버스 36종목 **전 종목 상시 구독**. on-demand 구독 관리는 YAGNI.

### 주문 체결: 서버 체결 필수
- 클라이언트가 보낸 가격은 절대 신뢰하지 않음. `POST /api/orders` → web이 워커 `/snapshot` 조회 → 그 가격으로 체결 → 트랜잭션(`orders` insert + `positions` upsert + 현금 차감).
- v1은 **시장가 즉시체결만**. 지정가는 미체결 상태머신 + 매칭 루프로 개발량 2배.
- 멱등성: 클라이언트 생성 `idempotencyKey`(unique 제약)로 더블클릭 방지.

### 포트폴리오 평가액
- **저장은 사실만, 계산은 조회 시점에**: DB에는 `qty`, `avgCost`, 실현손익만. 평가액·미실현손익은 클라이언트에서 스트림 가격으로 계산.
- 수익률 히스토리용 **일별 스냅샷**만 서버 기록(Vercel Cron 1일 1회).

### 인증
- **Auth.js(NextAuth v5) + GitHub/Google OAuth**, drizzle adapter. 이메일/비번 구현은 시간 낭비.
- 비로그인 둘러보기 허용, 주문·포트폴리오만 로그인 게이트.

### DB 스키마 뼈대 (6테이블 + auth)

```
users                id, name, email, image, createdAt
accounts_cash        userId PK, krw bigint, usd numeric(14,2)
orders               id, userId, market, symbol, side, qty numeric,
                     price numeric, filledAt, idempotencyKey UNIQUE
positions            (userId, market, symbol) PK, qty, avgCost, realizedPnl
watchlist_items      (userId, market, symbol) PK, createdAt
portfolio_snapshots  (userId, date) PK, totalValueKrw, cashKrw, cashUsd
instruments          market, symbol PK, name, currency, prevClose, prevCloseAt
```

돈은 전부 `numeric`(KRW는 bigint 가능), float 금지. US 소수점 주식은 numeric으로 허용(데모에 유리).

## 3. 함정과 리스크

1. **Vercel에서 WS/장시간 SSE 불가** — 현재 `/api/stream`은 Vercel에서 함수 실행시간 제한으로 끊기고, 연결 유지 시간만큼 과금, `globalThis` 싱글톤이 인스턴스마다 따로 생김. **로컬 개발용 폴백으로만 유지.**
2. **Finnhub 무료 티어** — REST 60콜/분, WS 1커넥션·심볼 ~50개. 저유동성 종목은 체결 없으면 틱도 안 옴 → 마지막 가격 + "체결 대기" 표시 필요.
3. **KIS가 최대 복병** — 실계좌 + 앱키 발급 필요. access token 24h 수명·재발급 1분 스로틀(캐시 필수), WS용 approval_key 별도, **실시간 등록 41건 한도**(체결가 18종목이면 OK, 호가까지 넣으면 초과), PINGPONG 응답 필수. 워커 코드 절반이 KIS 세션 관리가 될 것.
4. **하루 대부분 양쪽 장 마감** — KR 09:00~15:30, US 23:30~06:00 KST. 시장상태 배지 + 마지막 체결가 영속화(워커 재시작 시 증발 방지) + **데모용 replay/mock 모드 사실상 필수**.
5. **prevClose** — 현재 등락률은 전부 가짜. Finnhub `/quote`의 `pc`, KIS 일봉으로 매일 갱신 크론 필요. "전일" 기준 시각이 시장마다 달라 롤오버 로직 주의.
6. **환율** — 총자산 = KRW 현금 + KR 평가액 + (USD 현금 + US 평가액) × USDKRW. 무료 API 일 1회 캐시면 충분. 실시간 환율은 YAGNI.
7. **재연결 churn** — `usePrices`가 심볼 집합 변경마다 재연결하는 구조는 페이지 이동 잦으면 churn (v1 허용, 측정 후 개선).
8. **무료 호스팅 슬립** — Render Free는 15분 유휴 후 슬립 → 워커만큼은 $3~5 내고 상시 기동.

## 4. 기존 코드 재사용 평가

| 파일 | 판정 | 근거 |
|---|---|---|
| `lib/market/types.ts` | **유지** (worker와 공유로 승격) | Tick/Quote/keyOf 설계 깔끔 |
| `lib/market/format.ts` | **유지** | 통화 포맷·한국식 색상 규칙 그대로 |
| `lib/market/universe.ts` | **유지** → `instruments` seed로 | prevClose는 DB로 이관 |
| `lib/market/mock.ts` | **유지** (워커로 이동) | 키없음 모드·데모 replay에 계속 필요 |
| `lib/market/usePrices.ts` | **거의 유지** | `STREAM_URL` env 설계가 이미 워커 직결 상정 |
| `lib/market/priceSource.ts` | **폐기** (로직은 워커로 이식) | `globalThis` 싱글톤은 serverless에서 동작 불가 |
| `app/api/stream/route.ts` | **로컬 폴백으로 강등** | 배포 경로에서 제외 |
| `components/*` | **전부 유지** | 데이터 소스와 무관한 프레젠테이션 계층 |

버릴 코드가 사실상 `priceSource.ts` 한 파일뿐 — 기존 설계가 워커 분리를 이미 상정하고 있어 이식 비용이 낮다.

## 5. 오너에게 물어봐야 할 결정적 질문 5개

1. **현금 모델**: KRW 단일 지갑 + 자동 환전 vs 통화별 지갑 2개? → 지갑 2개가 압도적으로 단순 (주문 로직에서 환율 제거).
2. **주문 유형**: 시장가만 vs 지정가까지? → 지정가는 워커가 "중계기"에서 "체결 엔진"으로 격상, 개발량 2배.
3. **장 마감 시 동작**: 주문 차단 vs 마지막가 24시간 체결? replay 모드가 공식 요구사항인가?
4. **동접 규모**: <20(채용담당자) vs 수백(공개 서비스)? → <20이면 확장 코드 한 줄도 쓰지 말 것.
5. **KIS 실계좌·앱키 발급 의지**: 없으면 한국 주식은 mock 유지 or v2 연기 (지연시세 무료 소스는 사실상 없음).

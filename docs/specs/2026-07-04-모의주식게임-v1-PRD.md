## 1. 개요와 포지셔닝

### 1.1 한 줄 정의

**"매주 리셋되는 주식 배틀로얄 + 과거장 훈련소"** — 매주 월요일 전원이 가상 1,000만 원으로 리셋되어 실시간 시세(KIS 한국 + Finnhub 미국)로 경쟁하고, 장이 닫힌 시간에는 과거 실제 차트를 배속 재생하며 훈련하는 모의 주식 게임.

### 1.2 왜 이 조합인가

- **주간 시즌 리그**가 모의투자의 고질병(한 번 크게 잃으면 이탈, 고인물 복리 격차)을 구조적으로 해결한다. 실시간 시세가 '공정한 심판'이 된다.
- **리플레이 훈련소**가 실시간 모의투자의 최대 약점("장 열릴 때만 재밌다")을 해결한다. 장외·주말엔 홈이 리플레이 중심으로 전환되어 **24시간 죽지 않는 데모**가 된다.
- 두 모드는 분리되어 있다: 리플레이는 리그와 무관한 개인 훈련 모드이며, 성적은 프로필 기록만 남는다(확정 결정 #8).

### 1.3 제품 성격

- 1차 목적: **취업용 풀스택 포트폴리오** (풀스택 개발자 직군). 사용자 규모 목표 없음 — 데모 품질과 문서화가 우선.
- 운영 조건: 1인 개발(승윤), 예산 ~$5/월, 동접 <20 (면접관 + 친구 몇 명). **확장 코드 금지.**
- 이름 후보(미확정): 스톡아레나 / 개미리그 / 불장 / 리셋리그.
- 브랜드 무드: 웨불 네이비(딥네이비 + 시안 #22D3EE). 상승 빨강/하락 파랑 기본 + 설정 토글.

### 2.2 성공 지표 3개 — `docs/metrics.md`로 승격 (SQL 계산 가능)

아래 3개 지표는 쿼리째 `docs/metrics.md`에 상시 유지하며, PRD와 `docs/metrics.md`의 SQL은 완전히 동일하다. 봇(`is_bot`)과 익명 게스트(`is_anonymous`)는 활성화·리텐션 지표(①②)에서 제외한다. 표본 N<20이므로 전 지표는 **%와 절대 카운트를 병기**하고, 재참여율(②)은 정식 표본 확보 전까지 **env 단축 시즌 시뮬레이션 값**으로 기록한다.

**① 첫 주문 도달률** (활성화) — 목표 ≥ 60% (게스트 열람은 무료지만 첫 주문은 로그인+매매 결정 2단계 관문 — 초대 표본에서 과반 전환을 활성화 하한으로)

```sql
SELECT round(100.0 * count(DISTINCT o.user_id) / nullif(count(DISTINCT u.id), 0), 1) AS pct
FROM users u
LEFT JOIN orders o ON o.user_id = u.id AND o.status = 'filled'
WHERE u.is_bot = false AND u.is_anonymous = false;
```

**② 시즌 재참여율** (리텐션) — 목표 ≥ 40% (주간 리셋으로 복리 매몰비용이 없는 구조라 40% 복귀는 도전적 — 리셋·라이벌 루프가 재방문을 만드는지의 시금석)
prev/cur 모두 **orders 기준**(해당 시즌 실주문자)으로 확정한다. accounts는 lazy upsert(§4.4)로 첫 진입만 해도 생성되어 실매매 없는 계좌가 분모를 오염시키므로 배제한다.

```sql
WITH prev AS (SELECT DISTINCT o.user_id FROM orders o JOIN users u ON u.id = o.user_id
              WHERE o.season_id = :prev_season AND u.is_bot = false AND u.is_anonymous = false),
     cur  AS (SELECT DISTINCT o.user_id FROM orders o JOIN users u ON u.id = o.user_id
              WHERE o.season_id = :cur_season AND u.is_bot = false AND u.is_anonymous = false)
SELECT round(100.0 * count(*) FILTER (WHERE p.user_id IN (SELECT user_id FROM cur))
             / nullif(count(*), 0), 1) AS pct
FROM prev p;
```

**③ 리플레이 완주율** (콘텐츠 품질) — 목표 ≥ 70% (x30 배속 2~3분이라 이탈 비용이 낮음 — 시작자 다수가 끝까지 보는지를 콘텐츠 품질 하한으로)
(로그인 사용자 기준. 세션 시작 시 insert, 종료 시 `finished_at` 갱신 — 게스트는 기록하지 않으므로 분모에서 자연 제외)

```sql
SELECT round(100.0 * count(finished_at) / nullif(count(*), 0), 1) AS pct
FROM replay_sessions;
```

## 3. 페르소나와 핵심 동선

### 3.1 페르소나 A — 면접관 (기술 평가자)

- 40대 시니어 개발자 / 채용 담당. 데모 링크를 받고 **3분 내에 판단**한다. 계정을 만들 인내심이 없을 수 있다.
- **핵심 동선**: 링크 클릭 → **게스트로 즉시** 리더보드·탐색·종목상세 열람 (실시간 틱 확인) → 리플레이 1판 플레이(게스트 완전 허용) → 흥미가 생기면 OAuth 로그인 → 시장가 1건 체결 → 리더보드에서 자기 이름 확인.
- **설계 함의**: 게스트 게이트 최소화(§5.4), 장외 시간에도 살아있는 홈(리플레이 스위칭), README·ADR이 코드보다 먼저 보이는 리포지토리.

### 3.2 페르소나 B — 친구 참가자 (게이머형 사용자)

- 20~30대, 주식에 관심은 있으나 실계좌는 무섭다. 카톡으로 초대 링크를 받았다.
- **핵심 동선**: 링크 → 웰컴 모달("1,000만 원 지급 · 금요일 랭킹 확정") → 탐색에서 아는 종목 검색 → 첫 시장가 체결 → 체결 토스트 → 홈에서 내 순위 확인 → 주중 재방문(순위 확인) → 금요일 확정 결과 → 다음 주 월요일 리셋으로 재참여.
- **설계 함의**: 중간 합류 즉시 가능(§4.4), 현금 소진 시에도 다음 시즌 카운트다운 + 리플레이로 이탈 방지(§5.5), 온보딩 3종(§5.5).

## 4. 게임 규칙

### 4.1 시즌 수명주기

```
[월요일 KR 개장 전]           [주중]                [금 15:30 KST]        [확정~월요일]
   시즌 리셋 크론      →   실시간 매매·랭킹    →     시즌 확정 크론     →   결과 열람 기간
   · 기존 계좌 시드          · 지정가 매칭             ① open 주문 전량        · 신규 주문 차단
     1,000만 재설정          · 일별 스냅샷               expired + 예약          (seasons.status
   · 신규 시즌 row             (예약 현금 포함)          현금 환불               체크로 차단)
     생성(멱등)                                       ② finalValue 계산      · 홈: 결과 카드 +
                                                     ③ 랭킹·뱃지 확정          다음 시즌 D-N
```

- **시즌 경계 정합성 (시각 트리거 + 상태 기반 스윕)**: 확정은 15:30 크론 한 번에 의존하지 않는다 — 워커 부팅 시와 매 N분(env), `endsAt` 경과 & `status='active'`인 시즌을 스캔해 멱등 확정하고, 이미 `finalized`면 no-op다(크론이 지터·다운으로 누락돼도 다음 스윕이 복구). 확정의 **첫 스텝은 `seasons.status` CAS 플립**(`UPDATE ... SET status='finalized' WHERE id=$1 AND status='active'`) — 영향 행이 0이면 다른 스윕이 이미 처리한 것이라 전체 중단, 성공한 스윕만 이어서 **① open 주문 전량 `expired` 처리 + 예약 현금 환불 → ② finalValue 계산** 순서를 진행한다. 매칭 루프 스캔 쿼리에도 `seasons.status = 'active'` 조인이 들어가 확정된 시즌의 체결을 이중으로 막는다.
- **수동 재트리거 런북**: 확정 누락 의심 시 워커를 재기동하면 부팅 스윕이 밀린 시즌을 자동 처리한다(멱등이라 반복 안전).
- **확정 시각 정의**: 시즌 확정 = **금 15:30 KST 시점 스냅샷**. US 포지션은 그 시점의 마지막가(= 목요일 종가) 평가로 명시한다.
- **확정~월요일 리셋 사이**: 신규 주문은 `seasons.status` 체크로 차단하고, 홈은 §5.5의 상태 카드(다음 시즌 D-N + 리플레이 유도)를 재사용한다.
- **크론 검증 가능성**: 시즌 경계는 env로 파라미터화한다 — 로컬에서 "10분짜리 시즌"으로 리셋→매매→확정 풀사이클을 반복 테스트할 수 있어야 한다.
- **크론 내구성**: 리셋 크론은 멱등이며, 크론이 죽어도 첫 사용자 요청이 시즌 row를 lazy 생성해 복구한다(§7.6).

### 4.2 랭킹 산식

- 기본 지표: **시즌 수익률** `returnPct = (finalValue − seedMoney) / seedMoney`.
- **몰빵 방지**: 주문 서버 검증에 **종목당 매수 상한 = 시드의 40%** 를 둔다 → 최소 3종목 분산이 강제된다. (단일 종목 몰빵 도박으로 랭킹이 결정되는 것을 구조적으로 차단.)
- **동률 타이브레이커**: 시즌 **MDD가 낮은 쪽이 승리**. MDD는 `portfolio_snapshots`(§7.6에서 15:40 KST 1점) 기반이라 **주 5점 표본의 일별 근사**임을 명시한다(장중 저점은 포착하지 않음). `season_results.mdd`에 기록한다.
- `finalValue` = 확정 시점 현금(예약 환불 후) + Σ(보유 qty × 마지막가 × **환율**). 이때 곱하는 환율은 **확정 시점에 유효한 `fx_rates` 로우**(§6.6, 금 확정에 선행하는 09:00 KST 갱신분)로 통일한다.

### 4.3 봇 정책 — 공개 벤치마크

- 봇 3~5개, 전략: 랜덤 / 모멘텀 / 인덱스 홀딩. 리더보드 공백 방지 + **체결 엔진 상시 자가 테스트** 역할.
- 봇은 **BOT 배지 + 전략명 공개**(예: "인덱스봇"). 공식 순위·시즌 뱃지에서 제외(`WHERE is_bot = false`)하되 리더보드에는 **기준선으로 노출**한다 — "인덱스봇을 이겼는가"가 참가자의 자기평가 기준이 된다.
- **데모 요구사항**: 봇 매매 주기는 **"3분 관전 중 리더보드 순위 변동 ≥ 1회"** 를 기준으로 튜닝한다(§10 데모 스크립트의 전제 조건).
- **파라미터(env)**: 매매 주기(초 `BOT_INTERVAL_SEC`)와 1회 주문 시드 대비 비율(`BOT_ORDER_PCT`)은 env로 조정한다 — 데모 튜닝을 코드 수정 없이.
- **종목 선택 규칙**: 랜덤봇 = 유니버스 균등 추첨 / 모멘텀봇 = 직전 N분 등락률 상위 / 인덱스봇 = 시총 상위 종목 홀딩(리밸런싱 없음).
- **실행 위치·시장 게이트**: 봇은 **worker 루프에서 `fillOrder` 경유**로 주문한다(체결 엔진 자가 테스트 겸용). 시장 캘린더로 **열린 시장만** 거래하므로 주간 데모 시간대(장중 KR)엔 KR 봇이 활동한다.

### 4.4 시즌 중간 합류

- 첫 진입/첫 주문 시점에 현 시즌 계좌를 **lazy upsert**(시드 1,000만)한다. 월요일 크론은 기존 계좌 리셋만 담당한다.
- 리더보드에 즉시 등재하되 **"D+N 합류" 표기**로 공정성 맥락을 제공한다(`accounts.joined_at` 기반).

### 4.5 주말 챌린지

- v1은 시나리오가 1개(2020 코로나, §5.3)뿐이므로 주말 챌린지는 **동일 시나리오 내 서로 다른 2주 구간**을 잘라 '이번 주말의 챌린지'로 제공한다 — 추가 콘텐츠 비용 0. 콘텐츠 다양성을 주장하지 않으며(같은 역사 구간의 재편집), **시나리오 2개째를 확보한 뒤에만** 진짜 다양화로 확장한다.
- 리플레이 리더보드는 v1에서 만들지 않는다(§13 기각 기록 참조). 성적은 개인 프로필 기록만.

### 5.3 리플레이 훈련소 — 클라이언트 로컬 재생

- 시나리오 v1은 1개: **2020 코로나 폭락**. 일봉 정적 JSON을 리포지토리에 커밋(데이터 소스·라이선스는 §8).
- **전부 클라이언트에서 재생**: 정적 JSON + 클라이언트 타이머 배속 + 로컬 체결 계산. 서버는 세션 시작 insert / 종료 시 `replay_sessions` 갱신뿐. 워커의 feed 추상화는 **real/mock 2종**이며 replay는 포함하지 않는다.
- **심볼 큐레이션**: 리플레이는 실시간 유니버스 36종목과 **분리된 시나리오별 전용 심볼 세트**를 쓴다 — 해당 기간에 **이미 상장**되어 있고 일봉 데이터가 존재하는 종목만 큐레이션한다. 예: 2020 코로나 시나리오는 PLTR(2020-09 상장)·COIN(2021-04)·LG엔솔(2022-01)을 **제외**한다(사건 당시 미상장).
- 시나리오 데이터는 **week 1에 확보해 커밋**한다(리플레이는 절삭 불가이므로 데이터가 병목이 되면 안 됨). week 1 데이터 확보 완료 기준에 **심볼 커버리지 검증**(전 큐레이션 심볼이 시나리오 전 구간 일봉을 보유)을 포함한다.

### 5.4 게스트 정책

- **전면 공개**: 탐색, 종목상세(실시간가 포함), 리더보드 전체.
- **리플레이 플레이 완전 허용** — 결과 저장 시점에만 로그인 CTA. 게스트 결과는 insert 생략.
- **로그인 게이트는 주문 패널과 리그 참여뿐.** 게스트 홈 = 리더보드 + '참가하기' CTA.
- 인증: Google + GitHub OAuth (Better Auth v1.6, §8). 게스트는 Better Auth **anonymous 플러그인**으로 세션화.
- **게스트 승격**: anonymous 플러그인은 `isAnonymous=true`인 **실제 users 행**을 만들고, OAuth 링크 시 기본 동작으로 그 익명 행을 **삭제**한다. `onLinkAccount` 훅에서 데이터 이전을 수행하되, **v1은 게스트가 도메인 데이터(주문·계좌)를 만들지 않으므로 no-op**임을 명문화한다(게스트 게이트가 주문·리그 참여뿐이라 이전할 행이 없다).

### 5.5 온보딩과 상태 처리

- **온보딩 3종**: ① OAuth 콜백 후 보던 페이지로 복귀(callbackUrl) ② 첫 로그인 1회성 웰컴 모달 — "1,000만 원 지급 · 금요일 랭킹 확정" 2줄 + 탐색 CTA ③ 첫 체결 토스트 → 홈(내 순위) 유도.
- **현금 소진 상태**: 현금·포지션 모두 소진 시 홈 카드에 "다음 시즌 D-N + 리플레이 훈련소" 상태를 노출한다 — 파산이 이탈로 이어지지 않게 하는 핵심 루프. (시즌 확정~리셋 사이 기간에도 같은 카드 재사용.)
- **장외 지정가 피드백**: 장외 접수 시 "월 09:00 개장 후 체결 예정" 문구 + 주문함에서 open 주문 강조. 푸시 알림은 v2.
- **게스트 리플레이 결과 보존**: 게스트가 결과 저장(로그인 CTA)을 누르면 리플레이 결과를 **로그인 CTA 클릭 직전 `sessionStorage`에 직렬화**하고, OAuth 콜백 복귀 후 **멱등 재제출**(§6.1 idempotencyKey 규약 재사용)한다 — 왕복 중 결과 유실 방지.
- **스트림 끊김·백그라운드**: EventSource는 **명명 이벤트 소비**(§7.2) + 서버 하트비트 무수신 워치독(N초 초과 시 강제 재연결 + "연결 끊김" 배지 + 마지막가 유지)을 둔다. `visibilitychange`/`pagehide`에서 연결을 close하고 `pageshow`에서 재연결한다(모바일 백그라운드 좀비 연결·연결 상한 낭비 방지). **스테일(하트비트 끊김·30초 초과) 상태에선 시장가 버튼을 비활성**한다(§6.1 fail-closed와 일치).
- **저유동성 종목**: 마지막 체결가 + "체결 대기" 표시.

## 6. 주문·체결·정산 스펙

### 6.1 주문 수명주기

```
시장가:  접수 → [web이 워커 /snapshot 가격으로 즉시 체결] → filled
지정가:  접수(현금/수량 예약) → open → 워커 매칭 루프 가격 도달 → filled
                                  ├ 사용자 취소 → cancelled (예약 환불)
                                  └ 금 15:30 시즌 확정 → expired (예약 환불)
검증 실패(현금 부족·40% 상한·시세 스테일 등) → rejected
```

- **시장가**: web API가 워커 `/snapshot` 가격으로 즉시 체결. **클라이언트가 보낸 가격은 절대 신뢰하지 않는다.** 장 마감 중엔 시장가 차단(시장상태 배지), 지정가는 접수 허용 → 개장 시 매칭.
- **신뢰 경계**: 서버는 체결가뿐 아니라 **userId(세션에서)·seasonId(서버가 현재 active 시즌으로 결정)**도 클라이언트 입력을 신뢰하지 않는다. 클라이언트가 보내는 것은 `market·symbol·side·qty·limitPrice·idempotencyKey`뿐이다.
- **지정가**: 모의이므로 **유동성 무한 가정(가격 도달 시 항상 전량 체결)** 을 정책으로 명시한다.
- **멱등성**: `idempotencyKey`는 **`UNIQUE(user_id, idempotency_key)` 복합 제약**이며, 서버가 **UUIDv4 형식을 강제**한다. 중복 접수는 에러가 아니라 **원본 주문의 현재 결과를 반환**(멱등 재생)해 더블클릭·재시도·§5.5 재제출을 안전하게 흡수한다.
- **시세 신선도·워커 장애 게이트(fail-closed)**: 해당 시장 장중에 `/snapshot`의 `now − at > 30초`면 체결을 거부한다. **워커 응답 자체가 실패(연결 거부·타임아웃)해도** — web은 `/snapshot` 호출에 **2~3초 타임아웃**을 걸고 초과·실패 시 스테일과 **동일하게 fail-closed** — 동일 문구("시세 연결 복구 중")로 거부한다.

### 6.2 fillOrder 단일 함수 — 체결 로직 단일화

- 체결 트랜잭션(주문 상태 전이 + positions upsert + 현금 정산)은 `shared/`의 **단일 함수 `fillOrder()`** 로 만들고, **web(시장가)과 워커(지정가)가 공용**한다. 체결 규칙이 두 곳에서 미묘하게 달라지는 사고를 원천 차단.
- 워커는 `DATABASE_URL`을 공유해 Neon에 직접 쓴다(§7 다이어그램의 워커→Neon 화살표).
- **DB 드라이버**: 워커는 **node-postgres(`pg`) Pool**로 Neon에 직접 쓴다 — 상주 프로세스라 인터랙티브 트랜잭션(§6.3의 CAS+RETURNING 단일 트랜잭션)을 완전 지원한다. 장 마감 시 **pool을 해제**해 Neon autosuspend를 보존한다(§7.7). web은 서버리스/엣지에 맞는 기존 **`neon-serverless`**(WebSocket Pool)를 유지한다.

### 6.3 정확히-한-번 체결 (CAS)

- **모든 주문 상태 전이(fill·cancel·expire)는 `WHERE status='open'` CAS UPDATE ... RETURNING**이다. `fillOrder()`의 첫 문장 예:
  ```sql
  UPDATE orders SET status='filled', filled_price=$p, filled_at=now()
  WHERE id = $1 AND status = 'open'
  RETURNING reserved_krw;   -- 영향 행 0이면 전체 스킵
  ```
  취소·만료도 동일 패턴(`status='cancelled'|'expired' WHERE ... status='open' RETURNING reserved_krw`)이다. **환불액은 RETURNING으로 반환된 행에서만 파생**하고, 상태 전이·현금 환불·positions 정산 **전 과정을 단일 트랜잭션**으로 묶는다 → 취소 vs 확정 expire 경합, 크론 재실행에도 **이중 환불이 불가능**하다(전이를 이긴 쪽만 reserved_krw를 손에 쥔다).
- 현금 차감도 조건부 원자 UPDATE: `SET cash_krw = cash_krw - $x WHERE cash_krw >= $x` (영향 행 0 = 잔액 부족 = 롤백).
- **지정가 체결가 규칙**: 갭 통과 시 사용자에게 유리한 실제 도달 틱가로 체결한다 — **매수 = min(지정가, 도달 틱가)**, **매도 = max(지정가, 도달 틱가)**.
- **매수 지정가 차액 환급**: 매수 지정가 체결 시 예약액 `reserved_krw`와 실체결액의 차액(`reserved_krw − 실체결액`)을 같은 트랜잭션에서 **현금으로 환급**한다(예약은 지정가 기준, 체결은 더 유리한 틱가일 수 있으므로).
- **매칭 루프 신선도·세션 게이트**: 매칭 루프도 **30초 신선도 게이트 + 시장 캘린더 open 판정**을 경유한다 — 스테일 심볼(마지막 틱 30초 초과)은 스킵하고, 시장이 닫혀 있으면 매칭하지 않는다(§7.5).
- 매칭 루프는 인메모리 캐시를 진실로 삼지 않는다 — DB의 `status='open'`을 주기 재조회하며, 캐시가 어긋나도 CAS가 이중 체결을 막는다.

### 6.4 현금·주식 예약

- **매수 지정가**: 접수 트랜잭션에서 예상 금액을 **즉시 차감·예약**(`orders.reserved_krw`에 기록). US는 **접수 시점 fxRate를 고정 기록**해 예약액과 환불액이 항상 일치. 취소/만료 시 `reserved_krw`를 그대로 환불.
- **매도 지정가**: 접수 검증 = `보유 qty − 해당 종목 open 매도 주문 qty 합계 ≥ 주문 qty` (별도 reservedQty 컬럼 없이 open 주문 합산 서브쿼리로 처리 — 컬럼 동기화 버그 여지 제거).
- **종목당 40% 상한(TOCTOU 차단)**: 시드의 40% 매수 상한(§4.2)은 **접수 시 1차 검증 + 체결 트랜잭션 내부에서 재검증**한다. 재검증은 해당 종목의 **포지션·open 매수 주문을 `SELECT ... FOR UPDATE`로 잠근 뒤** 합산해 초과면 롤백한다 — 동시 접수된 두 주문이 각자 통과해 합산 40%를 넘기는 TOCTOU를 차단한다.
- **UI**: 매수 패널에 '주문 가능 현금'(예약 차감 후)을 표시하고, 부족 시 버튼 비활성.

### 6.5 costBasisKrw · realizedPnl 규칙

- positions는 주당 `avgCost` 대신 **총 취득원가 `costBasisKrw`**(KRW 환산 누계)를 저장한다. 매수 시 `costBasisKrw += 체결가 × qty × fxRate`(KR은 fxRate=1). 평단가가 필요하면 `costBasisKrw / qty`로 **파생 표시**한다.
- **매도 시**: `soldCost = costBasisKrw × (매도 qty / 보유 qty)`만큼 원가를 덜어내고(`costBasisKrw -= soldCost`, `qty -= 매도 qty`), `realizedPnl += 매도대금(KRW) − soldCost`. 즉 **realizedPnl = 매도대금(KRW) − 매도 수량 비례 원가**로 환차손익을 포함한다.
- 이 방식은 원가를 주당 반올림하지 않고 총액으로 덜어내므로 **라운딩 누적 없이 불변식 `Σ realizedPnl ≡ 현금 증감`이 정확히 성립**한다(정산 검증 테스트의 기준식).
- 금액 컬럼은 전부 `numeric`. **float 금지.**

### 6.6 환율

- KRW 단일 지갑 + US 주문 시 자동환전. **환율 시점 통일**: **매수 지정가만 접수 시점 fxRate를 고정**(예약액=환불액 정합, §6.4)하고, **시장가·모든 매도는 체결 시점** 환율을 주문 레코드 `fx_rate`에 기록한다(§6.5 realizedPnl 계산과 일치).
- 소스: **1순위 한국수출입은행 고시환율 API**(무료 키) + **폴백 frankfurter**. **일 1회 09:00 KST 크론 갱신**(금 15:30 시즌 확정에 선행해 확정이 항상 당일 유효 환율을 쓰게 함, §4.2), DB 단일 로우(`fx_rates`: rate, fetchedAt) 영속화.
- 빈 응답(주말·공휴일·오전 고시 전)이면 직전 값 유지. **48h 초과 스테일이면 UI에 스테일 배지.** fxRate 로우 자체가 없을 때만 US 주문 차단.

```
                │   ├─ POST /internal/orders/sync ←── 주문 upsert|cancel push(WORKER_SECRET) ── web
                │   └────────── fillOrder(지정가)·크론·마지막가 영속화 ──→ Neon Postgres
브라우저 ── 주문/리그 REST ──→ web (Next.js 16, Vercel 무료) ── fillOrder(시장가)·CRUD ──→ Neon (drizzle)
```

- **워커 push 계약**: web은 지정가 주문 접수/취소 시 워커에 `POST /internal/orders/sync`(헤더 `WORKER_SECRET` 필수, 페이로드 `{op: 'upsert'|'cancel', order}`)로 인메모리 open 주문 캐시를 갱신한다. **워커 무응답 시 web은 조용히 무시**한다 — 워커의 주기 재동기화 + §6.3 CAS가 안전망이라 push 유실이 정합성을 깨지 않는다.
- SSE는 **워커에 직결**(Vercel 경유 없음 — Vercel은 장시간 SSE 불가/과금). web은 인증·주문·리그 CRUD 전담.
- `shared/`: 타입(`Tick` 등 기존 `types.ts` 재사용) + `fillOrder()`. 별도 패키지 없이 파일 공유.
- 기존 코드 처분: `priceSource.ts`만 폐기(로직은 워커로 이식), `types/format/universe/mock/usePrices/컴포넌트` 전부 재사용, `app/api/stream`은 로컬 개발 폴백으로 강등.

### 7.2 SSE 프로토콜

- 연결(재연결 포함) 직후 워커가 `event: snapshot`으로 **시세북 전체 1건** 전송 → 이후 `event: ticks`로 델타 배치(1초). 재연결 시 스냅샷을 다시 받으므로 **Last-Event-ID 리플레이 버퍼 불필요**.
- **소비 계약**: 명명 이벤트이므로 클라이언트는 `addEventListener('snapshot')`(=시세북 **전체 대체**)와 `addEventListener('ticks')`(=델타 **병합**)로 각각 구독한다 — `onmessage`로는 명명 이벤트를 받지 못한다.
- **하트비트**: 서버는 **15초 주기로 SSE 코멘트(`:\n\n`) 하트비트**를 보내 유휴 연결 사멸을 막고, 클라이언트 무수신 워치독(§5.5)의 기준 신호가 된다.
- 스트림은 비인증 공개 + `Access-Control-Allow-Origin` = web 오리진 고정 + **동시 연결 수 상한** (동접 <20 전제의 안전핀).
- **fail-closed 부팅·연결 상한(T05 요건)**: 프로덕션에서 `WORKER_SECRET`·`CORS_ORIGIN`이 없으면 워커는 **부팅에 실패**한다(오구성으로 열린 스트림 방지). `/stream` **동시 연결 상한** 구현을 T05 완료 요건으로 명시한다.

### 7.3 /snapshot 계약

- 응답 스키마: `{symbol, price, at, source}`. web 주문 API는 §6.1의 30초 신선도 게이트를 적용.
- 워커는 부팅 시 `instruments.lastPrice/lastPriceAt`으로 시세북을 **워밍한 후** 서빙 시작(재시작 직후 빈 시세북 방지).
- web→worker 호출에는 `WORKER_SECRET` 헤더(양쪽 env).

### 7.4 feed 시장별 조립

- env 2개: `FEED_KR=kis|mock`, `FEED_US=finnhub|mock`. 시장별 독립 스왑 — KIS가 막혀도 US는 실시세 유지.
- 각 소스는 동일 `Tick` 시그니처 + `source` 필드. **mock 틱은 instruments 영속화에서 제외**, mock 시작가는 `prevClose`에서 시드(데모 등락률이 그럴듯하게 보이도록).

### 7.5 시장 캘린더 모듈

- IANA tz(`America/New_York`, `Asia/Seoul`) 기반 세션 판정 + 휴장일 정적 JSON(연 1회 갱신). **KST 절대시각 하드코딩 금지** — US 개장은 DST에 따라 22:30/23:30 KST로 변동.
- **날짜별 세션 오버라이드**: 캘린더 JSON에 날짜별 오버라이드 필드(임시공휴일 휴장, 수능일 KR 1시간 지연 개폐장, US 반일장 조기 마감 등)를 둔다. 정례가 아니므로 **발표 시 수동 패치**로 갱신하는 절차를 명시한다.
- **mock 시장은 항상 open**: `FEED_KR`/`FEED_US`가 `mock`인 시장은 세션 판정을 **open으로 간주**한다(데모·로컬 전용 규칙) — 장외 시간에도 §10 데모(가격+세션 게이트)를 재현하기 위함.
- **시장가 차단 · 지정가 매칭 루프 · lastPrice 갱신 3곳 모두 이 판정기를 경유**한다. 정규장 외(프리/애프터마켓) 틱은 폐기.

### 7.6 크론 설계 — 전부 워커의 node-cron

Vercel Cron은 사용하지 않는다(Hobby 2개 한도 + 최대 1시간 지터 + 실패 무감지 — 시즌 확정을 맡길 수 없음).

| 크론 | 시각(Asia/Seoul) | 내용 |
| --- | --- | --- |
| 시즌 리셋 | 월 KR 개장 전 | 기존 계좌 시드 재설정 + 신규 시즌 row(멱등, lazy 생성 폴백 있음) |
| 시즌 확정 | 금 15:30 + 부팅 시·매 N분 스윕 | `status` CAS 플립 → open 주문 expire + 환불 → finalValue → 랭킹·뱃지 (§4.1 순서 엄수, 멱등) |
| prevClose 갱신 | 07:30 | `instruments.prevCloseDate`로 멱등 처리 |
| 일별 스냅샷 | 15:40 | `portfolio_snapshots` 1점(당일 KR 종가 + 당일 새벽 US 종가 반영, MDD용 §4.2) |
| 환율 갱신 | 09:00 | §6.6 (금 15:30 확정에 선행) |

- **timezone 명시**: 모든 `node-cron` 등록에 `{ timezone: 'Asia/Seoul' }`를 명시한다 — node-cron 기본은 서버 로컬 tz이고 Railway는 UTC라, 명시하지 않으면 크론 시각이 9시간 어긋난다.
- **시즌 확정 스윕**: 확정은 15:30 크론 한 번에 의존하지 않고 워커 부팅 시·매 N분 상태 기반 스윕으로 멱등 확정한다(§4.1). **수동 재트리거**: 확정 누락 시 워커 재기동만으로 부팅 스윕이 밀린 시즌을 처리한다(멱등).
- 모든 크론은 **완료·실패 모두 Discord webhook 통지** (무감지 실패 방지·크론 하트비트 겸용 §11 R5, $0).

### 7.7 Neon 무료 티어 보존 패턴

- 워커의 DB 상시 연결은 금지 — 24/7 연결 시 **Neon Free 월 100 CU-hours**(2025-10 기준, 컴퓨트는 CU 가중이며 0.25~2 CU로 오토스케일)를 소진해 **DB 전면 정지** 리스크.
- **각성원 관리(awake time 최소화)**: ① **리더보드 엔드포인트에 30~60초 TTL 서버 캐시**(체결 시 무효화)로 폴링당 DB 히트를 흡수, ② **장외에는 시장 상태 기반으로 클라이언트 폴링을 중단**, ③ web `/api/health`는 **DB를 만지지 않는 정적 200 liveness**로 강등하고(§11 R5), DB·시즌 존재 같은 **readiness는 워커 크론 하트비트(Discord)**로 이관한다.
- 마지막가 영속화는 **장중 한정 60초 배치**. open 주문은 인메모리 캐시(기동 시 1회 로드 + web이 `POST /internal/orders/sync`로 접수/취소 push + 주기 재동기화, §6.3 CAS가 최종 안전망).
- 유휴 시 커넥션 해제로 Neon autosuspend 보존 — 워커는 **`pg` Pool을 장 마감 시 해제**(§6.2), web은 `neon-serverless`(WebSocket Pool, 트랜잭션 지원)를 유지.

## 9. DB 스키마 (최종)

10 코어 테이블 + `fx_rates` 단일 로우 + Better Auth adapter 테이블. 금액 전부 `numeric`, float 금지.

```
users                id, name, email, image, isBot boolean default false,
                     isAnonymous boolean default false,                     -- Better Auth anonymous 게스트(§2.2 지표 제외)
                     createdAt
                     (+ Better Auth adapter 테이블: session/account/verification)

seasons              id, startsAt, endsAt, seedMoney numeric, status
                     CHECK (status IN ('active','finalized'))

accounts             (userId, seasonId) PK, cashKrw numeric, joinedAt      -- "D+N 합류" 표기용
                     CHECK (cashKrw >= 0)

positions            (userId, seasonId, market, symbol) PK,
                     qty numeric, costBasisKrw numeric,                     -- 총 취득원가(KRW 누계), 평단가는 파생(§6.5)
                     realizedPnl numeric default 0
                     CHECK (qty >= 0)

orders               id, userId, seasonId, market, symbol, side, type,     -- type: market|limit
                     qty numeric, limitPrice numeric, filledPrice numeric,
                     fxRate numeric,                                        -- US: 매수 지정가=접수 시점 / 시장가·매도=체결 시점 고정(§6.6), KR: 1
                     reservedKrw numeric,                                   -- 매수 지정가 예약액(환불 원본)
                     status CHECK (status IN ('open','filled','cancelled','expired','rejected')),
                     idempotencyKey, createdAt, filledAt,
                     UNIQUE (userId, idempotencyKey),                       -- 복합 유니크 + 서버가 UUIDv4 강제(§6.1)
                     -- 매도 예약은 컬럼 없이 open 매도 주문 qty 합산 서브쿼리로 검증(§6.4)

season_results       (seasonId, userId) PK, rank, returnPct, finalValue, mdd  -- mdd = 타이브레이커

instruments          (market, symbol) PK, name, currency,
                     prevClose numeric, prevCloseDate date,                 -- 크론 멱등 키
                     lastPrice numeric, lastPriceAt                         -- 워커 재시작 워밍용

watchlist_items      (userId, market, symbol) PK, createdAt

portfolio_snapshots  (userId, seasonId, date) PK, totalValueKrw             -- 예약 현금 포함으로 정의(§4.1)

replay_sessions      id, userId, scenarioId, startedAt, finishedAt,         -- 완주율 = finishedAt 비율
                     returnPct, mdd

fx_rates             단일 로우: rate numeric, fetchedAt                     -- §6.6
```

**리더보드 데이터 경로**: web API가 전 참가자 `{userId, positions[], cashKrw, reservedKrw, fxRate}` 스냅샷을 단일 엔드포인트로 제공(reservedKrw = open 매수 주문 `reservedKrw` SUM 서브쿼리). 클라이언트가 구독 중인 SSE 가격(유니버스 36종목 전 구독)으로 전원 평가액을 로컬 재계산: `평가액 = cashKrw + reservedKrw + Σ(qty × SSE가격 × 환율)`. 폴링 refetchInterval 15~30초. 확장 코드 금지(<20 동접).
